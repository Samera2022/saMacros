# 付费功能网关实施与扩展指南

## 1. 背景与目标

当前项目已经将“脚本系统付费校验”从单点 UI 判断升级为 **core 内统一 Feature Gate**，目标是：

- 降低被单个 `if` 分支绕过的风险
- 让后续新增付费功能时复用同一套校验流程
- 保持 `samacros-api` 纯对外脚本开发接口，不承载授权语义

> 本文档用于后续持久化迭代，记录已完成内容、扩展方式与下一阶段改造方向。

### 2.5 完整性联动（已落地）

已实现“Rust + Java 双侧完整性探针”，并联动到授权放行决策：

- Rust：
	- 公钥分片重组
	- 分片哈希校验 + 总哈希校验
	- `samacros_license_integrity_probe` 导出探针
- Java：
	- 公钥字符串哈希校验
	- `LicenseIntegrityGuard` 汇总 Java/Rust 探针结果
	- `LicenseGateService` 在完整性失败时返回拒绝（`INTEGRITY_CHECK_FAILED`）

效果：

- 当关键完整性校验失败时，付费能力会被拒绝放行，而不是仅依赖 license 状态。

## 3. 设计边界（必须遵守）

### 3.1 `samacros-api` 不承载授权语义

`samacros-api` 仅用于脚本开发参考与公开 API，不放：

- 授权接口
- 许可状态模型
- 公钥、验签细节
- 任何可暴露商业能力边界的实现信息

### 3.2 授权逻辑归属

- **core**：业务网关编排与功能入口接入
- **rust-bridge（下一阶段）**：验签与信任根保护
- **gui**：仅负责展示与交互，不作为最终安全边界

---

## 4. 后续新增付费功能的标准接入流程

当未来新增一个需要付费的功能（例如“高级录制回放”）时，按以下流程执行：

### Step 1：定义能力点

在 `LicenseFeature` 新增枚举，例如：

- `ADVANCED_PLAYBACK`

### Step 2：配置授权映射

在 `LicenseGateService` 的 entitlement 映射中配置该能力：

- `PERSONAL` 是否允许
- `COMMERCIAL` 是否允许
- `ENTERPRISE` 是否允许
- `TRIAL` 是否允许（若允许可仅开放部分）

### Step 3：在 core 真实功能入口接入网关

在该功能实际执行入口调用：

- `LicenseGateService.checkFeature(LicenseFeature.ADVANCED_PLAYBACK)`

并按 `LicenseCheckResult` 决策：

- 允许：继续执行
- 拒绝：返回统一错误码/错误信息，禁止继续执行业务逻辑

### Step 4：GUI 只复用结果

GUI 层仅做用户提示：

- 展示购买弹窗或授权管理入口
- 不在 GUI 中实现“最终是否放行”的业务判断

---

## 5. 为什么这套方式可持续扩展

相较于“每个功能各写一套 if 校验”，统一网关有以下优势：

- 付费规则集中，不分散
- 新增功能仅需“加枚举 + 加映射 + 接入口”
- 便于后续替换底层实现（例如切换到 Rust 校验）
- 减少因 UI 绕过导致的直接执行风险

## 6. 下一阶段建议（P1）

当前网关仍是 Java 侧判定（基于 `LicenseManager`）。下一阶段建议：

1. 将“公钥信任根 + 验签”迁移到 `rust-bridge`
2. Java 仅获取能力快照/判定结果，不保留可篡改单点公钥常量
3. 增加短周期 lease（高价值功能）以降低被二进制注入后长期转卖风险

### 6.1 关于 keyring 的边界（必须明确）

`keyring` 是“多把受信公钥 + kid 选择规则”，适合做密钥轮换与灰度。

但它不能单独解决以下攻击：

- 攻击者在本地二进制/内存中替换某个 kid 对应公钥
- 攻击者直接 patch 掉验签分支

结论：

- `keyring` 是必要的工程能力，但不是充分安全条件
- 必须叠加“公钥隐藏化 + 完整性校验 + 运行时挑战”

### 6.2 公钥隐藏与完整性校验（推荐落地方案）

> 说明：公钥最终必须可用于验签，因此不能“不可逆加密后永不还原”。
> 正确做法是：用不可逆函数做承诺（commitment）与校验，而不是把公钥本体做不可恢复处理。

#### A. 公钥分片与延迟重组

- 不保存完整 PEM 常量
- 将公钥拆分为多片（建议 >= 5 片），分散在不同模块与函数中
- 每片使用不同掩码/包装（XOR + 常量扰动 + 运行时派生）
- 仅在验签瞬间于栈内存重组，完成后立即清零

#### B. 多重承诺校验（基于不可逆函数）

- 对最终公钥做 `SHA-256` 指纹承诺（至少两份，放在不同位置）
- 对每个分片再做独立承诺（分片级哈希）
- 运行时需同时通过：分片承诺 + 组装后总承诺
- 任一承诺失败即拒绝授权能力

#### C. 代码与资源完整性联动

- 验签前先做本地完整性检测：关键代码段/关键资源哈希
- 检测结果参与“是否放行付费能力”的决策
- 检测逻辑放在 Rust 与 Java 双侧，避免单点 patch

当前进度：

- ✅ 已完成“公钥材料完整性联动”
- ✅ 已完成“Rust + Java 双侧完整性探针联动授权放行”
- ✅ 已移除“产物哈希/构建指纹校验链路”
- ✅ 已移除“外置 integrity.properties 分发与自动加载逻辑”

当前完整性判定规则：

- 先执行 Java 侧公钥完整性校验（`OnlineActivationManager.isPublicKeyIntegrityValid()`）
- 在 Native Runtime 下再执行 Rust 侧完整性探针（`samacros_license_integrity_probe`）
- 任一失败则拒绝付费能力放行

当前推荐发布方式：

- 优先直接使用 `out/release/raw` 或 `out/release/upx` 目录进行分发
- 如需兼容旧流程，仍可使用 `out` 根目录文件
#### D. 多点校验而非单点 if

- 启动时校验一次（预热）
- 功能入口校验一次（Feature Gate）
- 高价值操作前再校验一次（执行前）
- 让攻击者必须同时绕过多处检查

#### E. 在线续租绑定（建议用于高价值功能）

- 本地长期 license 继续保留
- 增加短周期 lease（例如 12h/24h）
- lease 绑定：设备标识 + 功能集合 + 构建指纹 + 到期时间
- 风险环境（调试/篡改信号）下强制要求重新续租

### 6.3 关键认知

- 不可逆函数适合“验证是否被改”（哈希承诺），不适合“永久隐藏公钥且还要使用公钥”
- 对抗本地管理员/调试器的目标是“显著抬高攻击成本与维护成本”，而不是承诺绝对不可破解
