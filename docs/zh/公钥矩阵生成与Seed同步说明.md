# 公钥矩阵生成与 Seed 同步说明

## 1. 目的

当前项目已经把许可证公钥从“明文 / XOR 分片”迁移为“矩阵变换编码（PDR）”方案。

为了避免后续更换公钥时出现 **Java 与 Rust 两侧常量不同步** 的问题，仓库中新增了统一生成脚本：

- [/.scripts/gen_java_public_key.ps1](../../.scripts/gen_java_public_key.ps1)

这个脚本的职责是：

1. 读取 PEM 公钥
2. 生成矩阵编码常量
3. 输出 Java / Rust 所需的常量片段
4. 可选地自动同步 Java 与 Rust 中的 `EXPECTED_LOGIC_SEED`

---

## 2. 当前保护逻辑

### 2.1 Java 侧

位置：

- [samacros-core/src/main/java/io/github/samera2022/samacros/core/license/OnlineActivationManager.java](../../samacros-core/src/main/java/io/github/samera2022/samacros/core/license/OnlineActivationManager.java)

关键点：

- `KEY_M_INV` / `KEY_B` / `KEY_V_STORED`：矩阵编码后的公钥材料
- `EXPECTED_LOGIC_SEED`：由 `SHA-256(pem)[0..4]` 小端解释得到
- `isPublicKeyIntegrityValid()`：要求
  - 公钥可以成功解码
  - 计算出的 logic seed 与 `EXPECTED_LOGIC_SEED` 一致

### 2.2 Rust 侧

位置：

- [rust-bridge/src/license/mod.rs](../../rust-bridge/src/license/mod.rs)

关键点：

- `EXPECTED_LOGIC_SEED`
- `EXPECTED_XORCHECK`
- `integrity_probe()`：要求
  - PEM 可以成功组装
  - 计算出的 seed 与预期值一致
  - 运行时 `LOGIC_SEED / SEED_XORCHECK` 原子值与预期一致

---

## 3. 脚本输入与输出

## 3.1 输入

脚本输入是一份 PEM 公钥文件，例如：

```pem
<PEM_PUBLIC_KEY_HEADER>
...
<PEM_PUBLIC_KEY_FOOTER>
```

## 3.2 输出

默认会生成一个文本文件（默认位置为 `%TEMP%/pdr_matrix.txt`），其中包含以下分段：

- `SECTION:RUST_A`
- `SECTION:RUST_B`
- `SECTION:B`
- `SECTION:VSTORED`
- `SECTION:JAVA`
- `SECTION:SEED`

其中 `SECTION:SEED` 额外提供：

- `JAVA_EXPECTED_LOGIC_SEED=0x????????`
- `RUST_EXPECTED_LOGIC_SEED=0x????????`

---

## 4. 使用方式

## 4.1 仅生成常量文件

```powershell
powershell -ExecutionPolicy Bypass -File ./.scripts/gen_java_public_key.ps1 -PemPath <你的PEM文件路径>
```

示例：

```powershell
powershell -ExecutionPolicy Bypass -File ./.scripts/gen_java_public_key.ps1 -PemPath C:\Temp\public_key.pem -OutPath C:\Temp\pdr_matrix.txt
```

用途：

- 查看生成结果
- 手动检查矩阵常量
- 供后续脚本或人工比对使用

## 4.2 生成并同步 `EXPECTED_LOGIC_SEED`

```powershell
powershell -ExecutionPolicy Bypass -File ./.scripts/gen_java_public_key.ps1 -PemPath <你的PEM文件路径> -ApplyExpectedSeed
```

示例：

```powershell
powershell -ExecutionPolicy Bypass -File ./.scripts/gen_java_public_key.ps1 -PemPath C:\Temp\public_key.pem -OutPath C:\Temp\pdr_matrix.txt -ApplyExpectedSeed
```

用途：

- 在生成常量文件的同时
- 自动把 Java / Rust 两边的 `EXPECTED_LOGIC_SEED` 更新为同一个值

同步目标：

- [samacros-core/src/main/java/io/github/samera2022/samacros/core/license/OnlineActivationManager.java](../../samacros-core/src/main/java/io/github/samera2022/samacros/core/license/OnlineActivationManager.java)
- [rust-bridge/src/license/mod.rs](../../rust-bridge/src/license/mod.rs)

## 5. 推荐流程

当你需要替换许可证公钥时，建议按下面顺序执行：

### Step 1：准备新 PEM

把新的公钥保存为独立 PEM 文件。

### Step 2：生成矩阵常量并同步 seed

```powershell
powershell -ExecutionPolicy Bypass -File ./.scripts/gen_java_public_key.ps1 -PemPath <新PEM路径> -OutPath <输出路径> -ApplyExpectedSeed
```

### Step 3：把矩阵常量更新到代码中

当前脚本已经能自动同步 `EXPECTED_LOGIC_SEED`，但 `KEY_M_INV / KEY_B / KEY_V_STORED / Rust matrix rows / V_stored` 仍然是**输出到结果文件**，需要你按生成结果替换到代码中。

涉及位置通常包括：

- [samacros-core/src/main/java/io/github/samera2022/samacros/core/license/OnlineActivationManager.java](../../samacros-core/src/main/java/io/github/samera2022/samacros/core/license/OnlineActivationManager.java)
- [rust-bridge/src/license/matrix_rows_a.rs](../../rust-bridge/src/license/matrix_rows_a.rs)
- [rust-bridge/src/license/matrix_rows_b.rs](../../rust-bridge/src/license/matrix_rows_b.rs)
- [rust-bridge/src/license/mod.rs](../../rust-bridge/src/license/mod.rs)

### Step 4：编译验证

Java：

```powershell
mvn -pl samacros-core,samacros-gui -DskipTests compile
```

Rust：

```powershell
cd rust-bridge
cargo build --features quickjs
```

## 6. 为什么要同步 `EXPECTED_LOGIC_SEED`

如果只改了矩阵常量，但忘了改 Java 或 Rust 其中一边的 `EXPECTED_LOGIC_SEED`，会出现：

- Java 认为完整性失败，Rust 认为成功
- 或 Rust 认为完整性失败，Java 认为成功
- 最终导致授权探针结果不一致

因此 `EXPECTED_LOGIC_SEED` 应视为：

> 与 PEM、公钥矩阵材料强绑定的派生常量

它不应该手工维护两份，而应尽量由脚本统一生成。

---

## 7. 编码注意事项（重要）

在 Windows PowerShell 5.1 下：

- `Set-Content -Encoding UTF8` 会写入 **UTF-8 BOM**
- Java 源文件如果带 BOM，可能导致编译报错：`非法字符: '\ufeff'`

为避免这个问题，当前脚本内部已经使用 **UTF-8 无 BOM** 的写法回写源码。

如果你手工修改文件，也应避免引入 BOM。

---

## 8. 当前仓库中的自动同步锚点

Java：

```java
// AUTO-GENERATED by .scripts/gen_java_public_key.ps1 --ApplyExpectedSeed
private static final int EXPECTED_LOGIC_SEED = ...;
```

Rust：

```rust
/// AUTO-GENERATED by .scripts/gen_java_public_key.ps1 --ApplyExpectedSeed
const EXPECTED_LOGIC_SEED: u32 = ...;
```

只要这两个锚点还在，脚本就可以继续自动同步 seed。

---

## 9. 当前限制

当前脚本已支持：

- 生成全部矩阵常量到输出文件
- 自动同步 Java / Rust 的 `EXPECTED_LOGIC_SEED`

当前脚本**尚未直接自动回写**：

- Java 的 `KEY_M_INV / KEY_B / KEY_V_STORED`
- Rust 的 `matrix_rows_a.rs / matrix_rows_b.rs / key_vstored / key_offset_b`

也就是说，目前它是：

- **seed 自动同步**
- **矩阵常量半自动更新**

如果后续需要，可以再继续扩展为“一键全量回写”。

---

## 10. 一句话总结

现在推荐的做法是：

> 用 `gen_java_public_key.ps1` 作为 PEM 的统一生成入口，至少保证 Java / Rust 两侧的 `EXPECTED_LOGIC_SEED` 永远来自同一个公钥。
